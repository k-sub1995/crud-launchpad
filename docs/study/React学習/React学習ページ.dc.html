献立アプリで学ぶ React 入門

JavaScript → React 置き換えマップ

あなたが作った献立検索アプリ（recipe-app）のコードを題材に、Reactの主要概念を「今のコード」と比べながら学びます。左が今のJavaScript、右がReact。デモはどちらも実際に触れます。

画面が変わるまでの流れ — いちばん大事な違い
JavaScript イベント発生 → 自分でDOMを探す getElementById → 自分で書き換える innerHTML → 画面が変わる
React イベント発生 → stateを更新 setLikes(1) → Reactが差分を計算 → 必要な所だけDOM更新

Reactでは「画面 = stateから計算した結果」。 あなたが触るのはstateだけで、DOMの書き換えはReactの仕事になります。

01

JSX — HTML文字列の組み立てをやめる

今の src/ui/ui.js では、テンプレート文字列でHTMLを組み立てて innerHTML に流し込んでいます。Reactでは同じ見た目を JSX（JavaScriptの中に書けるタグ構文）で書きます。文字列ではないので、タグの閉じ忘れはエラーで教えてくれます。

JavaScript — 今のコード（ui.js）
function renderRecipeListItem(recipe) {
  // ただの「文字列」を組み立てている
  return `
    <article class="recipe-card">
      <a href="detail.html?id=${recipe.id}">
        <strong>${recipe.name}</strong>
      </a>
    </article>
  `;
}

// 使うとき: 文字列をDOMに流し込む
recipeList.innerHTML =
  recipes.map(renderRecipeListItem).join("\n");
React — これから
function RecipeCard({ recipe }) {
  // 文字列ではなく「JSX」を返す
  return (
    <article className="recipe-card">
      <a href={`/recipes/${recipe.id}`}>
        <strong>{recipe.name}</strong>
      </a>
    </article>
  );
}

// 使うとき: タグのように書くだけ
<RecipeCard recipe={recipe} />
JSXの約束ごと（3つだけ）:  ① class は className と書く  ② タグは必ず閉じる（<input />）  ③ { } の中にJavaScriptの式を書ける（${ } の代わり）
Q. JSXはHTMLの新しい書き方？
A. いいえ、JavaScriptの構文拡張です。ビルド時にただの関数呼び出しに変換されます。だから { } の中で変数・関数・三項演算子など、JSの式が全部使えます。
02

state — DOMを触るのをやめて、データを触る

レシピカードに「いいね」を付ける例です。JSでは addEventListener で登録し、データ（変数）とDOMの両方を自分で更新します。Reactでは useState のデータだけを更新すれば、画面はReactが合わせてくれます。イベントも onClick をタグに直接書きます。

JavaScript — 今のコード
// HTML側: あらかじめ id を振っておく
<button id="likeButton">♥ いいね</button>
<span id="likeCount">0</span>

// JS側: 探して、聞いて、自分で書き換える
let likes = 0;
document.getElementById("likeButton")
  .addEventListener("click", () => {
    likes++; // ① データを変えて…
    document.getElementById("likeCount")
      .textContent = likes; // ② DOMも自分で変える
  });
▶ 触ってみよう（JS版の動き）
オムライス
{{ likesJs }}
React — これから
function LikeButton() {
  // 「状態」を宣言する。likesが今の値、
  // setLikesが値を変える唯一の方法
  const [likes, setLikes] = useState(0);

  return (
    <button onClick={() => setLikes(likes + 1)}>
      ♥ いいね {likes}
    </button>
  );
  // DOM更新のコードはどこにもない。
  // stateが変わる→Reactがこの関数を再実行→画面更新
}
▶ 触ってみよう（React版の動き）
オムライス
Q. なぜ直接DOMを触ってはいけないの？
A. 触ってもいいのですが、アプリが育つと「データとDOMのズレ」がバグの温床になります。detail.js では保存のあとに renderRecipeDetail() → bindDetailControls() → toggleEditMode(false) と、DOMの整合性を保つ処理を自分で全部呼び直していますよね。Reactは「画面は常にstateから計算」なので、このズレが構造的に起きません。

Q. key って何のためにあるの？ index じゃだめ？
A. keyはReactが「前回とどの項目が同じか」を見分ける名札です。レシピアプリには recipe.id がすでにあるので、それをそのまま使えばOK。並び替えや削除があるリストでindexをkeyにすると、名札がズレて誤った再利用が起きます。

条件付きレンダリング — 「隠す」から「無くす」へ

詳細画面の編集モード切替です。今の detail.js は編集フォームを最初からHTMLに入れておき、display:none で隠しています。Reactは条件式で「描くか、描かないか」を決めます。切り替えながら、下の「DOMの状態」表示を見比べてください。

JavaScript — 今のコード（detail.js）
function toggleEditMode(show) {
  const editMode =
    document.getElementById("editMode");
  editMode.style.display =
    show ? "block" : "none";
}
// フォームは常にDOMの中にある。
// 「見えなくしている」だけ

Q. display:none と何が違うの？
A. JS版はフォームのHTML・入力値・イベントリスナーが常にメモリに居座ります。React版は不要なときDOMごと無い。表示条件が増えても「HTMLに全部入れて出し分ける」のではなく、条件式を書き足すだけで済みます。
05

フォーム — 値を「取りに行く」から「常に持っている」へ

レシピ登録フォームの例です。今の form-utils.js は、ボタンが押された瞬間に .value をDOMへ取りに行きます。Reactの制御コンポーネントでは入力値が常にstateにあるので、入力しながらのバリデーションや連動表示がタダで手に入ります。両方に文字を入れてみてください。

JavaScript — 今のコード（form-utils.js）
// 値はDOMの中。必要な瞬間に取りに行く
const text = document
  .getElementById("recipeText")
  .value.trim();

// リセットも自分でDOMを触る
recipeText.value = "";
recipeUrl.value = "";
▶ 入力してからボタンを押そう（JS版の動き）
JSが知っている値: {{ jsFormNote }}

入力中、JSは値を知りません。ボタンを押した瞬間にDOMへ聞きに行きます。

React — これから（制御コンポーネント）
const [name, setName] = useState("");

<input
  value={name}
  onChange={e => setName(e.target.value)}
/>

// 値は常にstateの中 → 連動表示も検証も簡単
<button disabled={name.trim() === ""}>追加</button>
// リセットは setName("") を呼ぶだけ
▶ 入力してみよう（React版の動き）
state: "{{ formReact }}"（{{ formReactLen }}文字）

1文字打つたびにstateが更新され、文字数もボタンの活性状態も連動します。

Q. 毎回stateを更新するなんて重くない？
A. 実用上は問題ありません。Reactは差分だけ更新するので、1文字の入力で変わるDOMはごくわずかです。それより「値がどこにあるか分からない」バグを減らせる利点の方がずっと大きいです。
06

useEffect — localStorage保存を「呼び忘れない」仕組みに

今の recipes.js は追加・更新・削除のそれぞれの関数の中で saveRecipes() を手動で呼んでいます。Reactの useEffect は「stateが変わったら自動で実行される処理」を1箇所に書けます。下のデモは本物のlocalStorageに保存しています — ページをリロードしても残るので試してみてください。

JavaScript — 今のコード（recipes.js）
export function addRecipe(recipe) {
  recipes.push(normalizeRecipe(recipe));
  saveRecipes(recipes); // ← 手動で保存
}
export function updateRecipe(id, recipe) {
  // …
  saveRecipes(recipes); // ← ここでも手動
}
export function deleteRecipe(id) {
  // …
  saveRecipes(recipes); // ← 呼び忘れたらバグ
}
▶ レシピ名を保存してみよう（JS版）
localStorageの中身: {{ lsJsRaw }}
React — これから
const [recipes, setRecipes] = useState(loadRecipes);

useEffect(() => {
  localStorage.setItem(
    "recipes", JSON.stringify(recipes));
}, [recipes]); // ← recipesが変わったら自動実行

// 追加でも更新でも削除でも、
// 保存のコードはこの1箇所だけ
▶ レシピ名を保存してみよう（React版）
localStorageの中身: {{ lsReactRaw }}
Q. useEffect はいつ動くの？
A. 画面の描画が終わった後です。第2引数の配列（依存配列）に入れた値が変わったときだけ再実行されます。[] なら初回だけ — 今の initializeRecipes()（起動時の読み込み）に相当します。
07

コンポーネントとprops — 画面を「部品」で組み立てる

index.html のトップ画面をReactで作るなら、こんな部品に分かれます。同じ RecipeCard 部品を、props（部品への引数）を変えて何度も使い回します。今のコードの renderRecipeListItem(recipe) の引数が、そのままpropsになるイメージです。

トップ画面の分解図
<App />
<Header /> 献立検索アプリ
<SearchBox />
材料または料理名…
検索
<RecipeList />
<RecipeCard recipe={オムライス} /> オムライス
<RecipeCard recipe={親子丼} /> 親子丼
<RecipeCard recipe={カレーライス} /> カレーライス
React — 部品の定義と利用
// 部品の定義: propsは「部品への引数」
function RecipeCard({ recipe }) {
  return (
    <article className="recipe-card">
      <strong>{recipe.name}</strong>
      <small>{recipe.ingredients.length} 材料</small>
    </article>
  );
}

// 組み立て: 親が子にpropsでデータを渡す
function App() {
  return (
    <>
      <Header />
      <SearchBox />
      <RecipeList>
        {recipes.map(r =>
          <RecipeCard key={r.id} recipe={r} />
        )}
      </RecipeList>
    </>
  );
}
データの流れは一方通行: 親 → 子 へpropsで渡すだけ。子は受け取ったpropsを変更できません。「誰がデータを持っているか」が常に明確になります。
Q. どこまで細かく部品に分ければいいの？
A. 最初は分けなくてOK。App.jsxに全部書く → 画面の大きな塊（Header / SearchBox / RecipeList）で分ける → 繰り返し出てくるもの（RecipeCard）を切り出す、の順で十分です。
08

総合演習 — 材料検索をReactで書き直すと

アプリの中心機能、材料検索です。ここまでの概念が全部登場します。注目ポイント: React版は検索結果をstateに入れません。結果は「入力stateから毎回計算するだけ」なので、検索ボタンすら不要になります（JS版は「卵」で検索ボタン、React版はただ入力してみてください）。

JavaScript — 今のコード（app.js 要約）
function searchRecipe() {
  // ① DOMから入力値を取りに行く
  const input = document
    .getElementById("ingredientInput").value;
  const terms = normalizeSearchInput(input);
  // ② 検索して…
  const results =
    findRecipesByIngredientsOrName(
      getRecipes(), terms);
  // ③ 結果をHTML文字列にして流し込む
  displaySearchResults(results, terms);
}
// ④ ボタンとEnterキーに自分で紐づける
searchButton.addEventListener("click", searchRecipe);
▶ 「卵」や「オムライス」で検索（JS版の動き）

検索ボタン（またはEnter）を押すと結果が表示されます。

{{ r.ing }}

React — これから
function SearchPage() {
  const [query, setQuery] = useState("");

  // 結果はstateに入れない。毎回計算するだけ
  const results = findRecipes(
    recipes, normalize(query));

  return (
    <>
      <input value={query}
        onChange={e => setQuery(e.target.value)} />
      <ul>
        {results.map(r =>
          <li key={r.id}>{r.name}</li>)}
      </ul>
    </>
  );
}
▶ 入力するだけで絞り込み（React版の動き）
{{ r.name }}
{{ r.ing }}

該当するレシピがありません。

Q. 検索結果もuseStateに入れるべき？
A. 入れません。入力（query）から計算できる値は「派生値」としてレンダリングのたびに計算します。stateを最小限にするのがReactの定石で、今のコードのグローバル変数 currentSearchIngredients のような「同期し忘れ」を防げます。
09

置き換え対応表 — 今のコードはこうなる

リプレース作業のチートシートです。左のパターンをコードで見つけたら、右の考え方に置き換えます。

JavaScript（今のコード）React（これから）
document.getElementById(...)→不要。JSXに直接書く（どうしても必要なら useRef）
el.innerHTML = `...` / textContent→JSXを返す。stateが変わればReactが更新
addEventListener("click", fn)→onClick={fn}
let recipes = [] などの可変変数→const [recipes, setRecipes] = useState([])
input.value を読む / 書く→制御コンポーネント（value + onChange）
style.display = "none" で出し分け→条件付きレンダリング（editing ? A : B）
map(...).join("\n") でHTML文字列化→map(r => <li key={r.id}>...</li>)
saveRecipes() を各所で手動呼び出し→useEffect(save, [recipes])
detail.html?id=1 へページ遷移→ルーティング（React Router / Next.jsで学ぶ）
renderRecipeListItem(recipe) 関数→コンポーネント <RecipeCard recipe={r} />
次のステップ — リプレースの進め方
① react-learning/ の環境で App.jsx に検索画面を全部書く → ② Header / SearchBox / RecipeList に分割 → ③ useEffect で localStorage 連携 → ④ 詳細・編集・削除 → ルーティングへ
