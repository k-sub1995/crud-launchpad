献立アプリで学ぶ React 入門

JavaScript → React 置き換えマップ

あなたが作った献立検索アプリ（recipe-app）のコードを題材に、Reactの主要概念を「今のコード」と比べながら学びます。
左が今のJavaScript、右がReact。
デモはどちらも実際に触れます。

画面が変わるまでの流れ — いちばん大事な違い
JavaScript イベント発生 → 自分でDOMを探す getElementById → 自分で書き換える innerHTML → 画面が変わる
React イベント発生 → stateを更新 setLikes(1) → Reactが差分を計算 → 必要な所だけDOM更新

Reactでは「画面 = stateから計算した結果」。 あなたが触るのはstateだけで、DOMの書き換えはReactの仕事になります。

01

JSX — HTML文字列の組み立てをやめる

今の src/ui/ui.js では、テンプレート文字列でHTMLを組み立てて innerHTML に流し込んでいます。
Reactでは同じ見た目を JSX（JavaScriptの中に書けるタグ構文）で書きます。
文字列ではないので、タグの閉じ忘れはエラーで教えてくれます。

JavaScript — 今のコード（ui.js） 
function renderRecipeListItem(recipe) {
  // ただの「文字列」を組み立てている
  return `
    <article class="recipe-card">
      <a href="detail.html?id=${recipe.id}">
        <strong>${recipe.name}</strong>
      </a>
    </article>
  `;
}

// 使うとき: 文字列をDOMに流し込む
recipeList.innerHTML =
  recipes.map(renderRecipeListItem).join("\n");
⚙ 実行時に起きること（JS）
① ${recipe.name} が埋め込まれた「ただの文字列」が完成する
→ ② innerHTML に代入
→ ③ ブラウザがその文字列をHTMLとして解析し直し、既存の中身を全部捨ててDOMを新規作成する
React — これから 
function RecipeCard({ recipe }) {
  // 文字列ではなく「JSX」を返す
  return (
    <article className="recipe-card">
      <a href={`/recipes/${recipe.id}`}>
        <strong>{recipe.name}</strong>
      </a>
    </article>
  );
}

// 使うとき: タグのように書くだけ
<RecipeCard recipe={recipe} />
⚙ 実行時に起きること（React）
① JSXはビルド時に関数呼び出しへ変換される（HTMLではない）
→ ② 実行すると「画面の設計図」となる軽いオブジェクトができる
→ ③ Reactが設計図と実際のDOMを比較し、違う箇所だけを反映する
JSXの約束ごと（3つだけ）:  ① class は className と書く  ② タグは必ず閉じる（<input />）  ③ { } の中にJavaScriptの式を書ける（${ } の代わり）
Q. JSXはHTMLの新しい書き方？
A. いいえ、JavaScriptの構文拡張です。
ビルド時にただの関数呼び出しに変換されます。
だから { } の中で変数・関数・三項演算子など、JSの式が全部使えます。
02

state — DOMを触るのをやめて、データを触る

レシピカードに「いいね」を付ける例です。
JSでは addEventListener で登録し、データ（変数）とDOMの両方を自分で更新します。
Reactでは useState のデータだけを更新すれば、画面はReactが合わせてくれます。

JavaScript — 今のコード 
// HTML側: あらかじめ id を振っておく
<button id="likeButton">♥ いいね</button>
<span id="likeCount">0</span>

// JS側: 探して、聞いて、自分で書き換える
let likes = 0;
document.getElementById("likeButton")
  .addEventListener("click", () => {
    likes++;
    document.getElementById("likeCount")
      .textContent = likes;
  });
⚙ クリックすると起きること（JS）
① クリック → 登録しておいたリスナー関数が呼ばれる
→ ② likes++ でデータを更新
→ ③ getElementById で表示先の要素を探す
→ ④ textContent を自分で書き換える。③④を書き忘れると、データは1なのに画面は0のまま（＝ズレバグ）
▶ 触ってみよう（JS版の動き）
オムライス
{{ likesJs }}
React — これから 
function LikeButton() {
  // 「状態」を宣言する。likesが今の値、
  // setLikesが値を変える唯一の方法
  const [likes, setLikes] = useState(0);

  return (
    <button onClick={() => setLikes(likes + 1)}>
      ♥ いいね {likes}
    </button>
  );
}
⚙ クリックすると起きること（React）
① クリック → onClick の関数が呼ばれる
→ ② setLikes(likes + 1) でstateの更新を依頼
→ ③ Reactが LikeButton() をもう一度実行（再レンダリング）
→ ④ 新旧のJSXを比較し、変わった数字のテキストだけDOMを更新。
DOM操作のコードは1行も書かない
▶ 触ってみよう（React版の動き）
オムライス
Q. なぜ直接DOMを触ってはいけないの？
A. 触ってもいいのですが、アプリが育つと「データとDOMのズレ」がバグの温床になります。
detail.js では保存のあとに renderRecipeDetail() → bindDetailControls() → toggleEditMode(false) と、DOMの整合性を保つ処理を自分で全部呼び直していますよね。
Reactは「画面は常にstateから計算」なので、このズレが構造的に起きません。

A. keyはReactが「前回とどの項目が同じか」を見分ける名札です。
レシピアプリには recipe.id がすでにあるので、それをそのまま使えばOK。
並び替えや削除があるリストでindexをkeyにすると、名札がズレて誤った再利用が起きます。

React — 部品の定義と利用 
// 部品の定義: propsは「部品への引数」
function RecipeCard({ recipe }) {
  return (
    <article className="recipe-card">
      <strong>{recipe.name}</strong>
      <small>{recipe.ingredients.length} 材料</small>
    </article>
  );
}

// 組み立て: 親が子にpropsでデータを渡す
function App() {
  return (
    <>
      <Header />
      <SearchBox />
      <RecipeList>
        {recipes.map(r =>
          <RecipeCard key={r.id} recipe={r} />
        )}
      </RecipeList>
    </>
  );
}
⚙ 実行時に起きること（React）
① App() が実行される
→ ② 中に書かれた子コンポーネントの分だけ、それぞれの関数が呼ばれる
→ ③ 以後、propsやstateが変わった部品だけが再実行される。
今の renderRecipeListItem(recipe) は「関数呼び出し＝部品化」の原型で、考え方はそのまま使える
データの流れは一方通行: 親 → 子 へpropsで渡すだけ。
子は受け取ったpropsを変更できません。
「誰がデータを持っているか」が常に明確になります。
Q. どこまで細かく部品に分ければいいの？
A. 最初は分けなくてOK。
App.jsxに全部書く → 画面の大きな塊（Header / SearchBox / RecipeList）で分ける → 繰り返し出てくるもの（RecipeCard）を切り出す、の順で十分です。

  const terms = query.trim().toLowerCase().split(/\s+/).filter(Boolean);
  const results = terms.length === 0
    ? defaultRecipes
    : defaultRecipes.filter(r =>
        terms.some(t =>
          r.name.toLowerCase().includes(t) ||
          r.ingredients.some(i => i.toLowerCase().includes(t))
        )
      );

  return (
    <div>
      <h1>レシピ検索</h1>
      <input
        value={query}                        // → 05章 制御コンポーネント
        onChange={e => setQuery(e.target.value)}
        placeholder="材料または料理名"
      />
      {results.length === 0 ? (            // → 04章 条件付きレンダリング
        <p>該当するレシピがありません。</p>
      ) : (
        <ul>
          {results.map(r => (               // → 03章 リストとkey
            <li key={r.id}>
              <strong>{r.name}</strong>（{r.ingredients.join("、")}）
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}

export default App;

ステップC — propsを深堀りする

Bで作ったSearchBoxは、実はもうpropsを使っています。
ここではJSの関数引数と何が違うのかをもう一段深く見ます。

JavaScript — 関数引数として渡す 
function renderRecipeListItem(recipe) {
  // この関数はrecipeを受け取って
  // 文字列を返すだけ。呼び出しっきりの
  // 使い捨てで、値の変化を検知できない
  return `<li>${recipe.name}</li>`;
}

// recipeが後で変わっても、この文字列は
// 自動的には更新されない
関数引数は「呼ばれた瞬間の値」を受け取るだけです。
呼び出しが終われば関数の仕事も終わり、渡した値が後で変わっても再実行されません。
画面を更新したければ、また同じ関数を手動で呼び直す必要があります。
React — props 
function SearchBox({ query, onChange }) {
  // queryが変わるたびにReactがこの関数を
  // 自動で再実行してくれる（見た目も最新化）
  return <input value={query} onChange={...} />;
}

// SearchBoxはqueryを書き換えられない。
// 変更は onChange 経由で親（App）に
// お願いするだけ = 一方通行
propsは「親から子への一方通行の入力」で、値が変わるたびにReactが子コンポーネントを自動で再実行します。
子はpropsを直接書き換えられません（読み取り専用）。
変更したいときは、親から渡された関数（onChange）を呼んで親にお願いする形にします。
見返してみよう: B章の <SearchBox query={query} onChange={setQuery} /> がまさにこの形です。
「誰が本当のデータ（state）を持っているか」が常にApp側の1箇所に定まるのは、この一方通行のおかげです。

ステップD — useStateを深堀りする

Aで書いた const [query, setQuery] = useState("") を例に、JSの普通の変数と何が違うのかをもう一段深く見ます。

JavaScript — ただの変数 
let query = "";

function onInput(e) {
  query = e.target.value;  // 変数は書き換わる
  // でも画面は何も変わらない。
  // 表示側を自分で呼ばないと反映されない
  renderResults(query);
}
変数はただの箱です。
値が変わったことを誰も見ていません。
画面を更新したければ、値を変えた場所ごとに「表示を作り直す関数」を自分で呼ぶ必要があります。
React — useState 
const [query, setQuery] = useState("");

function onInput(e) {
  setQuery(e.target.value);
  // これだけでOK。setQueryを呼ぶと
  // Reactが「再描画が必要」と認識し、
  // App関数をもう一度実行して画面を合わせる
}
useStateが返すqueryは「今のstate」のスナップショット、setQueryは「更新してもう一度描画して」とReactに依頼する関数です。
関数が再実行されても値が消えないのもポイントで、だから「コンポーネントのメモリ」と呼ばれます。
見返してみよう: A章で書いた setQuery(e.target.value) は、まさにこの「Reactへの再描画リクエスト」です。
02章の likes と全く同じ仕組みが、検索フォームでも使われています。

ステップE — useEffectで保存を自動化する

レシピ一覧を固定データからstateに昇格させ、追加フォーム（→ 05章）と自動保存（→ 06章）を付けます。
ここで登場する新機能が useEffect です。

JavaScript — 今の保存の仕組み
recipes.js では addRecipe / updateRecipe / deleteRecipe の3箇所それぞれで saveRecipes() を手動で呼んでいます。
「データを変えたら保存も呼ぶ」というルールを人間が覚えて守る方式——関数が増えるたび呼び忘れのリスクが増えます
React — useEffectとは
「このstateが変わったら、この処理を実行して」とReactに予約しておく仕組みです。useEffect(処理, [監視する値]) と書くと、描画が終わるたびにReactが監視値の変化をチェックし、変わっていたときだけ処理を実行。
保存・通信・タイマーなど「画面表示以外の仕事」の置き場所です

追加フォームと自動保存を足す

App関数の中身を以下のように変更します。

App.jsx の変更点 
import { useState, useEffect } from "react";  // ← useEffectを追加

function App() {
  const [query, setQuery] = useState("");

  // 起動時にlocalStorageから復元（無ければ初期データ） → 06章
  const [recipes, setRecipes] = useState(() => {
    const saved = localStorage.getItem("recipes");
    return saved ? JSON.parse(saved) : defaultRecipes;
  });

  // recipesが変わったら自動で保存。呼び忘れが起きない → 06章
  useEffect(() => {
    localStorage.setItem("recipes", JSON.stringify(recipes));
  }, [recipes]);

  // 追加フォーム用のstate → 05章
  const [newName, setNewName] = useState("");
  const addRecipe = () => {
    if (newName.trim() === "") return;
    setRecipes([...recipes,
      { id: Date.now(), name: newName.trim(), ingredients: [] }]);
    setNewName("");  // リセットもstateを変えるだけ
  };

  // resultsの計算は defaultRecipes → recipes に変えるだけ

  // returnに追加フォームを足す
  <div>
    <input value={newName}
      onChange={e => setNewName(e.target.value)}
      placeholder="新しいレシピ名" />
    <button onClick={addRecipe}
      disabled={newName.trim() === ""}>追加</button>
  </div>
}
✓ 動作確認
レシピを追加 → ブラウザをリロードしても消えないことを確認。
今の recipes.js では追加・更新・削除の3箇所で呼んでいた saveRecipes() が、useEffect 1箇所に集約されました。
