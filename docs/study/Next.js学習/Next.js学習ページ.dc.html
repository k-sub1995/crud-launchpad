献立アプリで学ぶ Next.js 入門

React → Next.js 置き換えマップ

React版レシピアプリを題材に、Next.jsの主要概念を比較して学びます。

ページが表示されるまでの流れ — いちばん大事な違い
React（今: CSR） ほぼ空のHTMLが届く → JSをダウンロード → ブラウザでReactが実行 → やっと画面が出る
Next.js（SSR） サーバーでReactを実行 → 完成済みHTMLが届く（即表示） → 後からJSが動き出す（ハイドレーション）

Next.js = 「Reactをどこで・いつ実行するか」を選べるようにしたフレームワーク。

コンポーネント・state・propsなどReactの知識はすべてそのまま使います。

新しく増えるのは「サーバーで動く部分」と「その境界線の引き方」です。

用語ディープダイブ（01–12）

この先の章で何度も出てくる7つの言葉を、具体的な処理の流れつきで先に1つずつ掘り下げます。

一気に読んでも、気になる言葉だけつまみ食いしてもOKです。

本編（01–08）
01

そもそもReactとは？ Next.jsとは？

この先ずっと「CSR」「SSR」のようなレンダリング方式の話が続きますが、その前に一番土台になる認識をそろえます。

ReactとNext.jsは対立する2つの技術ではなく、Next.jsはReactを内包する上位互換のフレームワークです。

React
UIを組み立てるための
「部品ライブラリ」

画面を「コンポーネント」という部品に分けて、propsとstateからHTML（見た目）を作ることだけに専念するライブラリです。

ルーティング、サーバーとの通信方法、ビルドの仕方などは決めていません。それらは自分で選んで用意する必要があります。

Next.js
Reactを内包した
「全部のせフレームワーク」

中身はそのままReactです。コンポーネント・props・stateの書き方は変わりません。

そのうえで、React単体には無かったルーティング・サーバー機能・画像最適化・ビルド設定などを標準搭載し、「Reactで画面を作る以外に必要なもの」を一式そろえています。

つまり

Next.js = React（部品を作る力）＋ルーティング・サーバー機能・最適化など（部品を届ける仕組み）

この2つ目の「届け方」の中でも、いちばん影響が大きいのが「HTMLをどこで・いつ組み立てるか（レンダリング方式）」の違いです。次のページからは、そこに絞って掘り下げます。

次のページでは、ReactとNext.jsの「結論」を先に一言でまとめます。

02

ReactとNext.jsの違い（レンダリング）

まず一番大事な結論だけ、先に押さえます。

この先の説明は、すべてこの1点の掘り下げです。

結論: 素のReactとNext.jsで「できること」が違う
素のReact（react-learning/）
CSRしかできない
HTMLを組み立てる場所はブラウザだけ。選択肢がない。
Next.js
CSRに加えて選べる
同じコンポーネントの書き方のまま、SSR・SSG・ISRもページごとに選べるようになる。

つまりNext.jsが足しているのは「新しい書き方」ではなく、「HTMLをいつ・どこで作るか」の選択肢です。

なぜReactは「ブラウザだけ」しか使えなかったの？

Reactはライブラリとして読み込まれているだけで、サーバー側のプログラムを書く仕組みがそもそもありません。

Next.jsはフレームワークとして「サーバー側で実行する場所（app/のフォルダ構成）」を用意しているため、同じReactのコンポーネントの書き方のまま、サーバー実行という選択肢が使えるようになります（01章で詳しく）。

次のページでは、そもそも「レンダリング」という作業自体を定義します。

03

レンダリングとは？

前ページで「CSR/SSRの違い」と言いましたが、そもそもレンダリングという言葉自体を定義します。

レンダリングとは、コンポーネントの中身（JSXやstate）を、実際にブラウザが表示できるHTML/DOMに変換する作業です。

<RecipeCard recipe={r} />
// ただのJSオブジェクト
レンダリング
→
<article>
 <strong>オムライス</strong>
</article>

「どこでこの変換計算をするか」がCSR/SSR/SSG/ISRの違いになります。この計算は2つの全く違うコンピューターのどちらでも行えます。

次のページでは、この「ブラウザ」と「サーバー」それぞれの特性を比べます。

04

ReactとNextのレンダリングの違いは？

レンダリング（＝HTMLへの変換）はブラウザとサーバーのどちらでも行えます。この「どこで」が両者の違いの正体です。

ブラウザ（クライアント）

ユーザー1人ぶんのマシンパワーを使う。
ネットワークが遅くても、一度JSが届けばその後は速く動ける。
ユーザーの手元にしかない情報（localStorageなど）に直接アクセスできる。
サーバー

開発者が管理する強いマシンで計算できる。
DBやAPIキーなど「秘密の情報」に安全にアクセスできる。
結果を使い回せる（同じHTMLを大勢に配れる＝SSG/ISR）。

「どちらで計算するか」によって、初回表示の速さ・サーバーの負荷・データの新しさ・秘密情報の扱いが変わります。

Next.jsで「サーバー行」が使えるようになる

該当なし
該当なし

Reactで使えるのは青の「CSR」のみ。Next.jsは「CSR」に加えて緑の「SSR」「SSG」「ISR」（サーバー行）も使えるようになります。

05

CSR — Client Side Rendering

ひとことで言うと: HTMLを組み立てる作業をブラウザの中でやる方式。

ReactはCSRしかできません。

▶ 具体的な処理の流れ — ユーザーがURLを開いてから
01
ブラウザが index.html をリクエストする
02
サーバーはほぼ空のHTML（<div id="root"></div>）とJSファイルのURLだけを返す — 画面はまだ真っ白
03
ブラウザがJSバンドル（Reactごと全部）をダウンロードし、パース・実行する
04
Reactが実行され、DOMを組み立ててここでやっと画面が表示される
05
以降のクリックや検索は、全部ブラウザ内のReactが処理する（サーバーへの依存なし）
具体例: 今のApp.jsx
// index.htmlの中身はこれだけ
<div id="root"></div>
<script src="/src/main.jsx"></script>

// main.jsx がブラウザで実行されて
// 初めてid="root"の中身が生まれる
ReactDOM.createRoot(
  document.getElementById("root")
).render(<App />);
向いている場面

ログイン後の管理画面・ダッシュボードなど「検索エンジンに拾われる必要がなく、開く人がログイン済みで限られる」画面。

向いていない場面

誰かに公開・共有するページ。初回表示の遅さとSEOの弱さが弱点になります（01章のQ&A参照）。
06

SSR — Server Side Rendering

ひとことで言うと: HTMLを組み立てる作業を、リクエストが来るたびにサーバー側でやる方式。

常に最新のデータでHTMLが作られます。

▶ 具体的な処理の流れ — ユーザーがURLを開いてから
01
ブラウザがURLをリクエスト
02
サーバーがその場でReactコンポーネントを実行し、DBやAPIからデータも取得してHTMLを組み立てる
03
完成済みHTMLが届き、この時点で内容が見える（JSはまだ届いていなくても文字や画像は表示済み）
04
裏でJSバンドルが届く
05
ハイドレーション（G5）が終わり、クリックや入力が効くようになる
具体例: Next.jsでの書き方
export default async function Page() {
  // cache:"no-store" = 毎回サーバーで
  // 新しく取得し直す → SSRになる
  const res = await fetch(
    "https://api/recipes/mine",
    { cache: "no-store" }
  );
  const myRecipes = await res.json();
  return <RecipeList recipes={myRecipes} />;
}
向いている場面

「ログインユーザーのマイレシピ」「在庫の残り数」など、アクセスするたびに中身が変わるページ。

コスト面の注意

アクセスのたびにサーバーが計算し直すため、SSG/ISRよりサーバー負荷が高く、表示もやや遅いのが弱点です。
07

SSG — Static Site Generation

ひとことで言うと: HTMLをデプロイする前に1回だけ作り置きしておく方式。

リクエスト時の計算がゼロなので、3方式の中で最速です。

▶ 具体的な処理の流れ
01
（デプロイ前） next build の実行時に、全レシピ分のHTMLを1つずつ生成しておく
02
作ったHTMLをCDNなどに配置してデプロイ完了
03
ユーザーがアクセス → 作り置きHTMLを即返すだけ（サーバーは何も計算しない、最速）
04
裏でJSが届き、ハイドレーションで操作可能に
05
レシピの中身を変えても、再ビルド・再デプロイするまで画面には反映されない
具体例: Next.jsでの書き方
export default async function Page({ params }) {
  // force-cache = ビルド時に1回だけ取得
  // してキャッシュを使い回す → SSG
  const res = await fetch(
    `https://api/recipes/${params.id}`,
    { cache: "force-cache" }
  );
  const recipe = await res.json();
  return <RecipeDetail recipe={recipe} />;
}
向いている場面

公開レシピの詳細ページのように、誰が見ても同じで、更新頻度が低いページ。

弱点

中身を更新するたびに再ビルドが必要。「新着レシピが増える一覧」のように更新はあるが都度SSRするほどではない場合は、次のISR（G4）が向いています。
08

ISR — Incremental Static Regeneration

ひとことで言うと: SSGの作り置きHTMLを、決めた時間ごとに裏側で作り直す方式。

SSGの速さと、SSRのような自動更新を両取りできます。

▶ 具体的な処理の流れ（例: 60秒ごとに再検証）
01
SSGと同じく、ビルド時（または初回アクセス時）に作り置きHTMLを用意
02
60秒以内のアクセスには、その作り置きHTMLを即返す（SSGと全く同じ速さ）
03
60秒を過ぎて誰かがアクセスすると、そのアクセスには古いHTMLを返しつつ、裏側で再生成を開始する
04
再生成が終わると、それ以降のアクセスから新しいHTMLに自動で切り替わる（再ビルド不要）

この「古いものを一旦返しつつ裏で更新する」考え方は stale-while-revalidate と呼ばれます。

具体例: Next.jsでの書き方
export default async function Page() {
  const res = await fetch(
    "https://api/recipes",
    // 60秒ごとに裏で再生成
    { next: { revalidate: 60 } }
  );
  const recipes = await res.json();
  return <RecipeList recipes={recipes} />;
}
向いている場面

「みんなのレシピ一覧」のように誰が見ても同じだが、新着が増えるページ。

SSR・SSGとの使い分け

「今すぐ確実に最新」が必須ならSSR、「多少の遅れは許容できる」ならISR、「ほぼ更新されない」ならSSG、が判断の目安です。
09

全体まとめ — ReactとNext.jsで選べる選択肢

05〜08でCSR・SSR・SSG・ISRを見てきました。ここで一度、全体を1枚にまとめます。

CSRSSRSSG / ISR
使える場所React / Next.jsNext.jsのみNext.jsのみ
計算する場所ブラウザサーバーサーバー

この先の読み進め方

レンダリングの選択肢はここまでで一通り揃いました。

この後の10〜12は、SSR/SSG/ISRを使うと必ず関わってくる「ハイドレーション」「ストリーミング」「RSCペイロード」という周辺の仕組みです。

読み終えたら本編01章から、実際にNext.jsのコードでこれらを使っていきます。

10

ハイドレーション（Hydration）

ひとことで言うと: サーバーが作った「見た目だけのHTML」に、ブラウザ側のReactが後からクリックやuseStateなどの「命」を吹き込む処理。

SSR・SSG・ISR、どの方式でも必ず1回発生します。

▶ 具体的な処理の流れ
01
サーバー製HTMLが届き、見た目だけは即座に表示される（この時点でボタンを押しても何も起きない）
02
裏でReact本体と各コンポーネントのJSがダウンロードされる
03
ReactがそのJSを実行し、「本来ならこういうDOMになるはず」という予想を組み立てる
04
予想と、実際に届いているサーバー製DOMを照合する
05
一致していれば、そのDOMにイベントリスナーとstateを紐付ける — ここでようやくクリックやuseStateが動き出す
⚠
一致していなければ → Hydration failed エラー
起きやすいミスの具体例
// サーバーでは実行時刻のA、ブラウザでは
// 数ミリ秒後の時刻Bになり、文字列が一致しない
<p>最終更新: {new Date().toLocaleTimeString()}</p>

// サーバーにはlocalStorageが存在しないので
// 常に「なし」判定 → ブラウザとズレる
<p>{localStorage.getItem("theme") ?? "デフォルト"}</p>
対処法

レンダリング中に「サーバーとブラウザで結果が変わりうる値」（現在時刻・乱数・localStorage・window幅など）を使わない。

どうしても必要なら、useEffect の中（＝ハイドレーション後、ブラウザ側だけ）で読み込みます。
11

ストリーミング / Suspense

ひとことで言うと: ページの中で遅いデータ取得がある部分だけを後回しにし、準備できた部分から先にHTMLを届ける仕組み。

「全部揃うまで真っ白」を防ぎます。

▶ 具体的な処理の流れ
01
レシピ詳細のうち「みんなの評価」だけ集計に時間がかかるとする。その部分を <Suspense> で囲む
02
サーバーは準備できたレシピ本体を先に送信開始（評価部分はまだ待たせない）
03
評価部分には一旦 fallback（スケルトンや「読み込み中」）が表示される
04
評価の集計が終わり次第、サーバーが追加でHTMLを送り、その部分だけ差し替わる（ページ全体の再読み込みなし）
具体例: Next.jsでの書き方
export default function RecipeDetailPage() {
  return (
    <article>
      <RecipeHeader />   ← 先に届く
      <Suspense fallback={<RatingSkeleton />}>
        <RatingSummary />  ← 遅れて届く
      </Suspense>
    </article>
  );
}
02章・05章との関係

フォルダに置く loading.tsx（05章）は、実はページ全体を自動でSuspenseで包む簡易版です。

ページの一部分だけを段階的に出したいときは、自分で <Suspense> を明示的に使います。
12

RSCペイロード — 遷移が速い裏側

ひとことで言うと: サーバーコンポーネントは、初回はHTMLとして届きますが、ページ内遷移のときは特殊な軽量データ形式（RSCペイロード）としてやり取りされます。

06章の <Link> がなぜ速いのか、の裏側です。

▶ 具体的な処理の流れ — 一覧ページから詳細ページへLinkで移動
01
<Link>が画面に見えた時点で、遷移先のRSCペイロードを先読み（prefetch）しておく
02
クリックすると、共通のlayout.tsx部分は再取得しない（すでに画面にあるものは使い回す）
03
先読み済みのRSCペイロード（差分だけの軽量データ）をReactが受け取り、ページ全体のHTMLではなく必要な部分だけを組み立て直す
04
画面が差し替わる — 通常のMPA遷移（JS版のdetail.html移動）のようなページ全体の再読み込みは起きない
なぜHTMLではなく専用データ形式を使うのか

HTML全体を毎回送ると、共通のヘッダーやレイアウトまで再送する無駄が出ます。

RSCペイロードは「サーバーコンポーネントの計算結果」を軽量なツリー構造で表現したもので、Reactはこれを受け取ってDOMのうち変わった部分だけを更新します。

普段のコーディングでこの形式を直接書くことはありませんが、「Linkが速いのはこの仕組みのおかげ」と知っておくと06章のprefetchが腹落ちします。
01

Next.jsとは — Reactに何が足されるのか

Vite + React の場合、

Viteは「開発サーバーとビルド」だけを担当し、ルーティングもデータ取得も自分で選んで足す必要があります。

Next.jsはそれらが最初から一体になったフルスタックフレームワークです。

React（Vite）— 今の構成
react-learning/
├── index.html      ← 唯一のHTML
├── vite.config.js
└── src/
    ├── main.jsx    ← ReactをDOMに接続
    └── App.jsx     

← 足りないライブラリは自分で追加:
//  ルーティング → react-router-dom
//  データ取得   → fetch + useEffect
//  サーバー処理 → 別途APIサーバーを立てる
⚙ 実行の仕組み（今）
すべてのコードはブラウザの中だけで動きます。

サーバーは index.html とJSファイルを配るだけの「ただの倉庫」です。

これをCSR（クライアントサイドレンダリング）と呼びます。
Next.js — これから
recipe-next/
├── next.config.js
└── app/            ← App Router
    ├── layout.tsx  ← 全ページ共通の枠
    ├── page.tsx    ← 「/」のページ
    └── recipes/
        └── [id]/
            └── page.tsx  ← 「/recipes/1」

// ルーティング → フォルダ構成がそのままURL
// データ取得   → コンポーネント内でawait
// サーバー処理 → 同じプロジェクトに書ける
⚙ 実行の仕組み（Next.js）
コードがサーバーとブラウザの2箇所で動きます。

初回表示はサーバーがHTMLを作って返し（SSR）、その後の操作はブラウザ側のReactが引き継ぎます。

「どのコードがどちらで動くか」を意識するのが、Next.js学習の中心テーマです。
Reactの知識はすべてそのまま:  コンポーネント・props・useState・useEffect・JSX・key、前作で学んだ全部をNext.jsでも使います。

Next.jsが変えるのは「書き方」ではなく「動く場所と動くタイミング」です。
用語のおさらい

CSR・SSR・SSG・ISR・ハイドレーション、ここから先の説明の前提になる言葉は目次冒頭のG1–G7で先に詳しく解説済みです。

ピンと来ない言葉があれば、サイドバーからいつでも戻れます。

Q. なぜCSRのままではだめなの？
A. だめではありません。

ただしCSRは「最初に空のHTMLが届く」ため、①初回表示が遅い（JSを全部落として実行するまで真っ白）、②検索エンジンやSNSカードが中身を読めないことがある、という弱点があります。

レシピを他人に公開・共有するアプリでは、この2つが効いてきます。
02

ファイルベースルーティング — フォルダ構成がURLになる

元のJS版は detail.html?id=1 とHTMLファイルを行き来し、React版ではReact Routerで <Route> を自分で並べる予定でした。

Next.jsでは設定ファイルすら不要。フォルダを掘るとURLが生えます。

下のデモでフォルダをクリックして、対応するURLを確かめてください。

React — Router設定を自分で書く
// URLとコンポーネントの対応表を
// JSXとして自分で管理する
<BrowserRouter>
  <Routes>
    <Route path="/"
      element={<SearchPage />} />
    <Route path="/recipes/:id"
      element={<DetailPage />} />
    <Route path="/register"
      element={<RegisterPage />} />
  </Routes>
</BrowserRouter>

// idの取り出しはフックで
const { id } = useParams();
⚙ 仕組み（React Router）
対応表はあくまでJSの実行結果です。

ページが増えるたびにこのファイルを編集し、パス文字列の打ち間違いは実行するまで気づけません。
Next.js — フォルダがそのまま対応表
app/
├── page.tsx              → /
├── register/
│   └── page.tsx          → /register
└── recipes/
    └── [id]/           ← 動的セグメント
        └── page.tsx      → /recipes/1, /recipes/2…

// idはpropsで受け取る（フック不要）
export default async function DetailPage(
  { params }
) {
  const { id } = await params;
}
⚙ 仕組み（Next.js）
page.tsx という名前のファイルを置くと、その場所がURLとして公開されます。

03

サーバー / クライアントコンポーネント — 一番の新概念

Next.jsのコンポーネントは、デフォルトで全部サーバーコンポーネントです。

サーバーで1回実行されてHTMLになり、ブラウザにはJSが送られません。

useStateやonClickを使いたい部品にだけ、先頭に "use client" と書いてクライアントコンポーネントにします。

「サーバー / クライアント」はどこで実行されるかの区別で、03章冒頭の用語辞典の「CSR/SSR」（いつ・どういう頻度でHTMLを作るか）とは別の軸です。

下のデモで、部品ごとにどちらにすべきか試してみてください。

React — 全部クライアントで動く
// 今のApp.jsx: 何も宣言しなくても
// 全コードがブラウザで動く
function App() {
  const [query, setQuery] = useState("");
  // stateもイベントも自由に使える。
  // そのかわり全コンポーネントのJSが
  // ブラウザに送られる
}
⚙ 今の動き
「サーバーかクライアントか」という区別自体が存在しません。

アプリが大きくなるほど、ブラウザに送るJSの量が増え続けます。
Next.js — 境界線を自分で引く
// app/page.tsx — 宣言なし = サーバー
// HTMLだけ届く。JSは送られない
export default function Page() {
  return <RecipeList recipes={recipes} />;
}

// SearchBox.tsx — 対話が必要な部品だけ
"use client";  // ← この1行が境界線
import { useState } from "react";

export function SearchBox() {
  const [query, setQuery] = useState("");
  // useState/onClickはここでだけ使える
}
⚙ Next.jsの動き
サーバーコンポーネントはサーバーで実行後、結果のHTMLだけが届きます。

"use client" を書いた部品（とそのimport先）だけJSが送られ、ブラウザで動き続けます。

境界を最小にするほどページは軽くなります。
04

"use client" が届く範囲

前ページで「対話が必要な部品だけに "use client" をつける」と書きましたが、その指示がどこまで効くのか、よくある誤解を含めて掘り下げます。

"use client" が届く範囲 — 誤解しやすいポイント

"use client" はそのファイル1つだけの印ではありません。

そのファイルがimportしている子コンポーネントすべても、まとめてクライアントバンドルに引き込まれます（これを「クライアント境界」と呼びます）。

✗ ありがちな失敗

ページの一番外側（詳細ページ全体など）に "use client" を書いてしまう。

→ そのページ配下の全部品がクライアントに引き込まれ、SSRの利点（軽さ）が消える。
✓ 良い書き方

「編集ボタン」「検索欄」など末端の小さな部品だけに "use client" を付け、そこをページ側からimportして埋め込む。

→ 引き込まれる範囲を最小化できる。
"use server" — 逆方向のもう1つの宣言
// app/actions.ts — Server Action
"use server";  // ← この関数はサーバーでだけ実行

export async function addRecipe(formData) {
  const name = formData.get("name");
  await db.recipes.insert({ name }); // DB直操作OK
}

// クライアントコンポーネントのフォームから
// 直接この関数を呼べる（API不要）
<form action={addRecipe}>
  <input name="name" />
  <button>追加</button>
</form>
"use client" とは向きが逆です。

"use client" は「ここから先はブラウザで動かして」というサーバー→クライアントの境界線。

"use server" は「この関数だけはブラウザから呼ばれてもサーバー側で実行して」というクライアント→サーバーの入口（＝Server Action）です。

07章の「Server Actions」はこの仕組みのことで、フォーム送信時にfetchやAPI Routeを自分で書く必要がなくなります。
▶ レシピアプリの部品を仕分けしてみよう — サーバー？クライアント？

部品を選んで「サーバー」「クライアント」どちらかを押すと、正解と理由が出ます。

「{{ quizName }}」はどっち？
{{ quizVerdict }}
{{ quizReason }}
⚠ ハマりどころNo.1: サーバーコンポーネントで useState を使うと You're importing a component that needs useState... というエラーになります。

対処は「その部品を小さく切り出して "use client" を付ける」。

ページ全体をクライアントにするのは最終手段です。
05

データ取得 — useEffectの「後から取る」をやめる

レシピをDBやAPIから取ってくる場面です。

Reactでは「まず空で描画 → useEffectで取得 → stateに入れて再描画」という3拍子でした。

Next.jsのサーバーコンポーネントはコンポーネント自体をasyncにして、描画の前にawaitで取得します。

React — useEffectで後から取得
function RecipeList() {
  // ① まず「空」の状態で描画される
  const [recipes, setRecipes] = useState([]);
  const [loading, setLoading] = useState(true);

  // ② 描画後にuseEffectで取得
  useEffect(() => {
    fetch("/api/recipes")
      .then(res => res.json())
      .then(data => {
        setRecipes(data); // ③ 再描画
        setLoading(false);
      });
  }, []);

  // ローディング表示も自分で管理
  if (loading) return <p>読み込み中…</p>;
  return <ul>…</ul>;
}
⚙ 画面に起きること
① 空のリストで1回描画される（チラつき） → ② 通信 → ③ データが入って再描画。

loading・エラーのstate管理、通信の重複実行対策など、定型コードが毎回必要です。
Next.js — サーバーで取得してから描画
// app/page.tsx（サーバーコンポーネント）
export default async function Page() {
  // 描画の前にサーバーで取得が終わる。
  // DBに直接アクセスしてもOK
  const recipes = await getRecipes();

  // データが揃った状態のHTMLが届く
  return (
    <ul>
      {recipes.map(r =>
        <li key={r.id}>{r.name}</li>)}
    </ul>
  );
}

// 読み込み中表示は loading.tsx を
// 置くだけ（stateでの管理が不要）
⚙ 画面に起きること
ブラウザにはデータ入りの完成HTMLが最初から届きます。

空表示のチラつきも、loading用stateも、useEffectの定型文も消えます。

02章の詳細ページなら「URLのidでawait取得 → そのまま描画」の2行です。
Q. じゃあuseEffectはもう使わない？
A. 出番は減りますが、なくなりません。

「初回のデータ取得」はサーバーコンポーネントに移りますが、ブラウザでしかできないこと（localStorage、タイマー、ブラウザAPIの購読など）はこれからも "use client" な部品の中でuseEffectを使います。

前作06章の知識はそのまま有効です。
06

レンダリング方式 — いつHTMLを作るかを選ぶ

Next.js最大の武器は「HTMLをいつ作るか」をページごとに選べることです。

下のデモで方式を切り替えて、タイムラインの違いを見比べてください。

ハイドレーションって結局なに？

サーバーが返すHTMLは見た目だけで、まだクリックしても何も起きません。

その直後、ブラウザに届いたJSがそのHTMLに対して「ここがボタン、ここにこのイベントを付ける」と後付けで結びつける作業がハイドレーションです。

これが終わって初めて、useStateやonClickが実際に動き出します。SSR・SSGどちらの方式でも、この工程は必ず発生します。
▶ 方式を切り替えてみよう — ユーザーがURLを開いてからの流れ
{{ ts.num }}
{{ ts.text }}
レシピアプリでの使いどころ: {{ renderUseCase }}
選び方の目安:  データが誰でも同じ＆ほぼ更新されない → SSG。

リクエストごとに中身が変わる（ログインユーザー別など） → SSR。

誰でも同じだが時々更新される（一覧ページなど） → ISR。

Next.jsではfetchのキャッシュ設定などで方式が決まり、同じコードのまま切り替えられます。
⚠ ハマりどころ: ハイドレーションエラー
「サーバーが作ったHTML」と「ブラウザでReactが計算した結果」が食い違うと Hydration failed エラーになります。

典型例は new Date() や乱数、そしてlocalStorage読み込みをレンダリング中に使うこと（サーバーとブラウザで結果が違うため）。

localStorageに触るのは、必ずuseEffectの中（＝ブラウザ側の描画後）にします。
07

Linkとレイアウト — ページ間で共有するもの

複数ページになると「共通のヘッダー」と「ページ移動」の扱いが必要になります。

元のJS版では全HTMLファイルに同じ<header>をコピペしていました。

Next.jsでは layout.tsx と <Link> が担当します。

React（+Router）— 自分で組む
// 共通レイアウトも1つのコンポーネント
// として自分で設計して全Routeを包む
<BrowserRouter>
  <Header />   ← 自分で置く
  <Routes>…</Routes>
</BrowserRouter>

// 移動は react-router の Link
<Link to={`/recipes/${r.id}`}>
  {r.name}
</Link>
⚙ 仕組み
クリックするとRouterがURLを書き換え、対応するコンポーネントに差し替えます（ページ再読み込みなし）。

ただし遷移先のコードやデータの先読みは自分で工夫する必要があります。
Next.js — layout.tsx と Link
// app/layout.tsx — 全ページ共通の枠。
// childrenに各page.tsxがはまる
export default function RootLayout(
  { children }  // ← 前作07章のpropsそのもの
) {
  return (
    <html lang="ja">
      <body>
        <Header />
        {children}   ← ここが差し替わる
      </body>
    </html>
  );
}

// 移動は next/link の Link
<Link href={`/recipes/${r.id}`}>
  {r.name}
</Link>
⚙ 仕組み
layout.tsxはページ遷移しても再レンダリングされず、中身（children）だけ差し替わります。

さらに <Link> は画面内に見えた時点で遷移先を自動で先読み（prefetch）するため、クリック時にはほぼ待ちゼロで表示されます。
⚠ ハマりどころ: 普通の <a> タグを使ってしまう
見た目は同じでも <a href="/recipes/1"> はページ全体を再読み込みし、stateも消えます（元のJS版と同じMPA遷移に戻ってしまう）。

アプリ内の移動は必ず <Link>、外部サイトへは <a>、と使い分けます。
08

総合 — レシピアプリのNext.js版 全体構成

ここまでの概念を組み合わせると、レシピアプリはこういう構成になります。

ポイントは「ほとんどサーバー、対話する部品だけクライアント」という境界の引き方です。

ファイル構成と境界線
app/
├── layout.tsx        🖥 サーバー 共通ヘッダー
├── page.tsx          🖥 サーバー レシピ一覧を取得
│   └── SearchBox     🌐 クライアント 検索入力(useState)
├── recipes/[id]/
│   ├── page.tsx      🖥 サーバー idで1件取得
│   │   ├── EditForm  🌐 クライアント 編集モード(useState)
│   │   └── DeleteBtn 🌐 クライアント 確認モーダル(useState)
│   └── loading.tsx   🖥 サーバー 読み込み中表示
└── register/
    └── page.tsx      🖥 サーバー
        └── Form      🌐 クライアント 登録フォーム
データの持ち方も変わる

JS版・React版ではlocalStorageがデータの保管庫でした。

Next.jsではサーバーがあるので、データはサーバー側（DBやファイル）に置くのが基本形になります。

登録・編集・削除は「Server Actions」というサーバー側の関数をフォームから直接呼ぶ形になり、「API作成 → fetch → state更新」の手間が大幅に減ります。
前作との対応

・検索の絞り込み（08章の派生値）→ SearchBoxの中でそのまま
・編集モード切替（04章）→ EditFormの中でそのまま
・useEffect保存（06章）→ Server Actionsに置き換え

Reactで学んだ書き方は部品の中で全部生きています。
Q. localStorageのままではだめ？
A. 動きますが、Next.jsの利点が半減します。

localStorageはブラウザごとの保存領域なので、サーバーはレシピの中身を知れず、SSRしても空のHTMLしか作れません（05章のハイドレーション問題も起きやすい）。

学習の順番としては「まずlocalStorageのまま移植 → 動いたらサーバー側に移す」の2段階がおすすめです。

⚠ 注意すべき点 まとめ（ハマりどころ TOP4）
1. "use client" の付け忘れ / 付けすぎ
useState・onClickを使う部品には必須。

4. サーバーコンポーネントでブラウザAPIを触る
